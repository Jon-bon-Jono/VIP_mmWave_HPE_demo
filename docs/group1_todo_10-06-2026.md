# Group 1 TODO list

This list gives Group 1 students useful ways to enter the project while live radar integration is still on hold. The first task for everyone is to set up the environment, run the current replay demo, and understand the current code structure. After that, students can consider the contribution areas below.

These tasks are intentionally open-ended. Students should not treat them as fixed implementation instructions. 
---

## 0. Required starting task: setup and code walkthrough

Before choosing a contribution area, each student should:

- set up the Conda/ROS 2 environment;
- build the workspace;
- run the current replay demo;
- understand the main ROS nodes, packages, topics, and data flow;
- read the root `README.md` and `docs/ros2_dev_notes.md`.

Students should be able to explain, at a high level, how the current replay demo works:

```text
replay pickle data
  -> radar replay node
  -> /radar/points
  -> pose inference emulator or pose replay node
  -> /pose/people
  -> visualiser
```

They should also understand the broad roles of:

```text
vip_hpe_core      shared Python utilities, constants, and preprocessing
vip_hpe_msgs      custom ROS 2 message definitions
vip_hpe_runtime   ROS 2 nodes, launch files, visualisation, replay runtime
```

## 1. Logging, debugging, and runtime observability

The runtime system should become easier to debug while it is running. Students working on this task should investigate how each ROS node currently reports what it is doing, and propose improvements that are easy to manage from the session configuration file.

Many students may not be familiar with logging verbosity. A simple way to think about it is that there are multiple tiers of logged information ranked based on importance/detail:

```text
ERROR    something failed and probably needs attention
WARNING  something unexpected happened, but the system may continue
INFO     useful high-level status, such as node startup and selected config
DEBUG    detailed information useful while developing or diagnosing a problem
```

The system should eventually support different levels of terminal output depending on the chosen verbosity. For example, a normal demo session might only configure the runtime system to show important startup/status messages in the terminal, while a debugging session might show more detailed per-frame information. 

However, terminal output is not the whole story. As the project moves toward live multi-modal data collection, timing and traceability will become important. It may be useful for each runtime session to save **all** logged information to a file, even if only a smaller subset is shown in the terminal. Logs should include an operating-system timestamp so that they can later be compared with other data streams, hardware events, or external recording systems.

Questions to explore:

- What should each node log when it starts?
- What should be logged every frame, every N frames, only in debug mode, or only when something goes wrong?
- How should logging verbosity be configured from the session YAML file?
- What timestamp information should be included in each log entry?
- Are there current failures that are too silent or hard to understand?
- How should a user know which session config, input file, topics, and parameters are active?

Possible contributions:

- improve startup logs for one or more nodes;
- propose a clean logging configuration section for the session YAML;
- add optional debug logs that can be enabled or disabled;
- make warnings and error messages clearer;
- investigate ROS 2 logging conventions and recommend how this project should use them;
- prototype saving runtime logs to a per-session file.

A good outcome is not necessarily a large logging framework. A good outcome could simply be that the next student can run the demo and immediately understand what each node is doing, while still leaving room for more systematic session logging later as this system expands.

## 2. Radar hardware and configurable point-cloud generation parameters

This task is for students interested in the radar hardware side of the project. Live radar integration is not ready yet, but students can still make progress by reading the Texas Instruments documentation and understanding the configurable parts of the 3D people-counting / point-cloud generation pipeline.

The focus is not to write the driver yet. The focus is to understand what the radar demo is doing and how configuration choices could affect the point clouds we receive.

Texas Instruments documentation is notoriously dense. Not all information presented the documentation is necessary for this project. As a first step, it is important to first determine what you need to take away from documents before you start to digest the material (ChatGPT can come in handy at this first step). The following docs will be a useful starting point:
* [3D People Counting Demo Software Implementation Guide](https://e2e.ti.com/cfs-file/__key/communityserver-discussions-components-files/1023/3D_5F00_people_5F00_counting_5F00_demo_5F00_implementation_5F00_guide.pdf)
* [Group Tracker Parameter Tuning Guide for the 3D People Counting Demo](https://e2e.ti.com/cfs-file/__key/communityserver-discussions-components-files/1023/3D_5F00_people_5F00_counting_5F00_tracker_5F00_layer_5F00_tuning_5F00_guide.pdf)
* [People Tracking and Counting Reference Design Using mmWave Radar Sensor](https://www.ti.com/lit/ug/tidue71d/tidue71d.pdf?ts=1781002981869)

Questions to explore:

- Which parameters can be configured in the TI 3D people-counting demo?
- Which parameters affect the radar front-end, chirp configuration, frame configuration, point-cloud generation, filtering, detection, or tracking?
- Which parameters affect range resolution, velocity resolution, angular resolution, maximum range, field of view, frame rate, point density, noise level, or track stability?
- Which parameters are likely to differ between public training datasets and our future live radar setup?
- How could these differences create a domain gap between training data and live data?
- Which parameters should we record as metadata whenever we collect live data?
- Which parameters are important enough that ML students should know about them?
- Since we plan to use the MM-Fi dataset to train our model, do the authors give any hints about the configuration of their point cloud generation pipeline in the [paper](https://arxiv.org/pdf/2305.10345) or [codebase](https://github.com/ybhbingo/MMFi_dataset/tree/main)?

Possible contributions:

- summarise the configurable parameters in the TI people-counting demo;
- identify which parameters are most likely to cause domain shift and ultimately affect ML model performance;
- create a short radar-configuration glossary for the team;
- propose what radar configuration metadata should be saved with future recordings;
- identify risks for training on one radar configuration and deploying on another.

A good outcome would be a practical note that helps the rest of the team understand how radar configuration choices may affect the data distribution seen by the model.

## 3. Front-end redesign: beyond matplotlib

The current matplotlib visualiser is useful for demonstrating the ROS replay pipeline, but it is probably not the best long-term front end. Students working on this task should investigate alternatives and recommend a direction.

This should not be treated as a one-way visualisation problem only. A future front end may need to act as a control interface for the runtime system. For example, the user may need buttons or controls to:

- pause and resume a session;
- start, stop, or reset a session;
- inspect current runtime status;
- select or confirm a session configuration;
- view warnings or errors;
- interact with replay or live data collection.

The task is not simply “replace matplotlib immediately”. The task is to understand what the front end needs to do, compare possible approaches, and prototype something small if appropriate.

Questions to explore:

- What does the front end need to show?
- What does the front end need to control?
- Is the front end mainly for debugging, demonstration, annotation, evaluation, live monitoring, data collection, or some combination of these?
- What are the weaknesses of the current matplotlib visualiser?
- Should we use a ROS-native tool such as RViz2?
- Should we use Foxglove Studio or another ROS-compatible visualisation/control tool?
- Would a custom Python viewer using PyQtGraph, VisPy, Open3D, Qt, or another library be better?
- Which options are practical on Windows?
- Which options are easiest for future students to install and use?
- Which options can show point clouds, skeletons, tracks, IDs, timestamps, and debugging overlays?
- Which options can also provide runtime controls such as pause/play/start/stop/reset?

Possible contributions:

- compare several front-end options;
- test whether one option can subscribe to or display the current ROS topics;
- investigate whether one option can also provide runtime controls;
- prototype a minimal alternative viewer or control panel;
- propose what message formats, services, actions, or adapter nodes would be needed for a better front end;
- recommend whether matplotlib should remain as a debug-only viewer.

A good outcome is a clear recommendation with evidence, not necessarily a finished replacement.

## 4. Pose topology and skeleton-format flexibility

The current visualiser was initially designed around the Kinect/Living Lab 32-joint pose format. Future datasets and models may use different pose topologies, such as the MM-Fi 17-joint format or other custom skeleton layouts.

Students working on this task should investigate how pose topology is currently assumed in the code and propose a flexible way to support multiple skeleton formats.

Questions to explore:

- Where does the current code assume Kinect’s 32-joint topology?
- What would happen if the pose message contained 17 joints instead of 32?
- How do different datasets define different joint names and skeleton edges?
- Should topology be defined in code, in config files, in constants, or in message metadata? It should probably be attached to configuration files for the HPE model itself (these don't exist yet, but they may function similar to the point cloud preprocessor, e.g. dummy_pointcloud.yaml). It a pose topology configuration should probably also be attached to a pose replay node aswell. In the future we will probably want to replay data collected from models which produce different pose topologies. 
- What is the simplest useful abstraction for pose topology?
- How can the visualiser draw skeletons without hard-coding one dataset’s joint layout?
- How should missing joints, unknown joints, or different joint orderings be handled?

Possible contributions:

- document where topology assumptions currently exist;
- propose a minimal design for multiple pose topologies;
- implement a small proof of concept;
- add tests or simple examples for more than one topology;
- make the visualiser fail gracefully when it receives an unsupported pose format.

A good outcome is a lightweight design that makes future pose formats easier to support, without over-engineering a large class hierarchy too early.

## 5. Session records and saved runtime information

Each time the runtime system is launched, it may be useful to create a session record. This is separate from the session configuration file. The configuration file describes what should run; the session record describes what actually happened during a particular run.

For now, this feature may only need to save logged information. In the future, it may need to support multi-modal data collection, including radar streams, camera streams, model outputs, annotations, timing information, device information, and configuration metadata.

This part of the system should therefore be modular, clean, and extensible. It should not be a quick one-off hack that only works for the current replay demo.

Questions to explore:

- What information should be saved for each runtime session?
- Where should session outputs be stored?
- How should each session be named or identified?
- Should the system save a copy of the session YAML used for that run?
- Should the system save logs, warnings, errors, node startup information, and runtime metadata?
- How should OS timestamps and ROS timestamps be represented?
- What would need to change later when live radar, camera, or other sensors are added?
- Should session recording be managed by one node, by launch code, by a utility module, or by the future front end?

Possible contributions:

- propose a clean session-record structure;
- prototype saving logs and metadata for each runtime launch;
- investigate how session records should interact with the session YAML config;
- recommend how future multi-modal recordings should be organised;
- identify what should be saved now so that future debugging and dataset collection become easier.

A good outcome is a small, extensible session-recording concept that works for the current demo but does not block future multi-modal data collection.

## 6. Home/landing page for runtime session configuration

FYI: this is a currently a lower priority task. It would be best to attack this task once we have a better idea about how we will integrate the radar into the system. 

In the long term, users may not always launch the runtime system directly from the command line and they may not configure a session by modifying a YAML file. We may need a simple home or landing page that helps users configure a session before starting the runtime system.

Developers can still launch the system directly by passing YAML files, because this is faster during development. The home/landing page is mainly for making the runtime system easier to use once live radar hardware and multi-modal data collection are introduced.

For example, once radar hardware is added, the page may need to ask which COM port the radar is connected to, what radar configuration should be used, whether data should be saved, and which front-end mode should run. The output of the home/landing page could be a session configuration file similar to the current session YAML files which can be passed to the session builders currently being invoked in `src/vip_hpe_runtime/launch/session.launch/py`'s `_launch_setup` function.

Questions to explore:

- What information does a user need to provide before starting a session?
- Which settings should remain developer-only YAML options, and which should be exposed through a user interface?
- How should the page detect or ask where the radar is connected?
- How should radar COM port, radar configuration, replay/live mode, logging settings, and session output settings be represented in YAML?
- Should this page be part of the same front end as the visualiser, or a separate pre-launch configuration tool?
- Should it generate YAML only, or also start/stop the runtime system?
- How can we avoid making the UI too rigid while the runtime architecture is still changing?

Possible contributions:

- sketch a simple home/landing page design;
- identify what session settings should be configurable through the UI;
- prototype a minimal YAML-generating configuration page;
- propose how the UI should represent radar connection information;
- recommend whether this should be a web page, desktop GUI, ROS tool, or something else.

A good outcome is a practical design or small prototype that helps us think about future runtime usability without forcing developers to stop using YAML directly.

## 7. Student-proposed contributions

Students may propose their own small contribution after they understand the current system.

A good proposed task should:

- be useful even while live radar integration is on hold;
- help the student understand the system better;
- be small enough to review;
- avoid large architecture changes unless discussed first;
- produce something that helps development, testing, debugging, visualisation, documentation, or future data collection.

Possible examples:

- a small unit test for a fragile function;
- a debugging utility for checking ROS topics;
- a profiling script for visualisation or preprocessing;
- clearer error handling for one failure mode;
- a small code cleanup that removes duplication;
- a short technical note on a design choice;
- a comparison of possible libraries or tools;
- a small proof of concept for a future feature.

Students should discuss their proposed contribution before spending too much time implementing it.

## Suggested contribution style

Students should prefer small, reviewable contributions over large rewrites.

A useful contribution might be:

- a short investigation note;
- a small bug fix;
- a small test;
- a small prototype;
- a clear recommendation;
- a small improvement to logging, usability, or debugging.

Students should avoid:

- rewriting major parts of the architecture without discussion;
- making large changes that are hard to review;
- coupling `vip_hpe_core` to ROS, PyTorch, or matplotlib;
- adding dependencies without checking whether they are necessary;
- spending too long on a task before asking for feedback.

The main goal is to learn the system and make small improvements that help the project move forward.
